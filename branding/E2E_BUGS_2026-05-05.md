## Kritisch (vor Pioneer-Launch zwingend zu fixen)

### Bug #7 — TypeError im Init blockt halbe App (kritisch, gefixt)
**Status:** ✅ gefixt (Commit folgt)
**Schwere:** kritisch
**Wo:** clean.html init beim Login
**Symptom:** Console-Error `TypeError: Cannot read properties of null (reading 'addEventListener')` bei Line 9123 (`$('setShareLoc').addEventListener`). Crasht die ganze Init-Sequence danach — wahrscheinlich Ursache für leeren Mood-Picker (Bug #3) und vermutlich auch Onboarding-Tour-Skip.
**Root cause:** `setShareLoc`-Element existiert nicht mehr im HTML (wurde im Privacy-Pane Sprint G zu `setShareLive` umbenannt), aber zwei dangling JS-Blöcke griffen noch auf `$('setShareLoc')` zu — ohne Null-Check.
**Fix:** Beide Blöcke entfernt bzw. mit `if ($('setShareLoc'))` Null-Check geschützt. JS-Validation läuft, kein dangling-ID mehr (alle 74 addEventListener-Targets matchen IDs im HTML).

### Bug #1 — Supabase Auto-Confirm umgeht Email-Verify
**Status:** offen
**Schwere:** kritisch
**Wo:** Sign-Up-Flow
**Symptom:** Sprint-2.5 implementiert einen Email-Verify-Banner der erscheinen soll wenn `!email_confirmed_at`. In der Realität setzt Supabase `email_confirmed_at` direkt beim Sign-Up (gleiche Sekunde wie `created_at`), weil im Project das Auth-Setting „Confirm email" deaktiviert ist.
**Fix:** Supabase Dashboard → Authentication → Providers → Email → Toggle „Confirm email" auf ON. Optional: Trigger der `email_confirmed_at` zurücksetzt für neue Sign-Ups bis User klickt.
**Gegencheck nach Fix:** Neuer Sign-Up → muss in Verify-Banner landen, Login-Versuch ohne Klick muss blockieren.

### Bug #2 — Onboarding-Tour wird nicht ausgelöst
**Status:** offen
**Schwere:** hoch
**Wo:** Erster Login nach Sign-Up
**Symptom:** Direkt nach Sign-Up landet User im Home-Tab — die Welcome-Tour aus Sprint 3.3 (6 Slides mit „Niemand geht alleine heim." etc.) zeigt sich nicht.
**Vermutung:** `maybeStartOnboarding()` checkt auf `!profile.onboarding_completed_at && !profile.name`. Beim Insert wird `name: u.email.split('@')[0]` gesetzt → name ist nicht null → Bedingung false. Die Tour startet nie.
**Fix:** Bedingung auf `!profile.onboarding_completed_at` reduzieren. Oder: Default-Name aus dem Insert raus, damit `name` null bleibt bis zum ersten Save.
